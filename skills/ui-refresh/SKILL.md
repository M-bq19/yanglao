# UI 美化技能（ui-refresh）

## 一、技能用途

本技能用于处理以下类型任务：

- 遗留项目页面太旧、太乱、太厚重
- 前后台风格不统一
- 希望只改页面层，不改运行环境
- 希望分阶段推进，而不是一次性大改
- 希望先找公共框架，再做页面细化，再做统一收口

本技能特别适用于当前这种项目：

- `back/` 和 `front/` 分目录
- 后端仍在 IDEA 中启动
- 前端仍在 VSCode 中启动
- 目标是统一美化，而不是架构重构

---

## 二、技能适用前提

使用本技能前，默认满足以下前提：

1. 项目已经可以运行
2. 前后端目录结构明确
3. 当前任务以视觉和交互层优化为主
4. 不允许随意改数据库、接口、依赖、启动方式
5. 页面较多，必须分阶段推进

---

## 三、技能核心目标

通过低风险方式完成以下目标：

1. 找到前后台各自的公共布局与公共样式
2. 找到菜单配置、首页入口、列表模板、表单模板
3. 先统一后台公共框架
4. 再统一前台公共框架
5. 再对典型页面逐页细化
6. 最后统一颜色、字体、按钮、边框、阴影、圆角、表单、卡片、分页等视觉规范

---

## 四、必须遵守的非谈判约束

使用本技能时，默认必须遵守：

- 不新增依赖
- 不修改 `pom.xml`
- 不修改 `package.json`
- 不升级 Spring Boot / Vue / Element / Layui / jQuery
- 不修改接口返回结构
- 不修改数据库结构
- 不重写业务逻辑
- 不改变当前启动方式
- 不擅自迁移框架
- 不为了美化去重构核心业务

---

## 五、标准工作流

### 第一步：项目扫描

先做全局扫描，只读不改：

1. 找根目录说明文件
2. 找后端公共布局
3. 找后端菜单配置
4. 找后端公共 CSS
5. 找前台首页入口
6. 找前台公共样式
7. 找前台列表页模板
8. 找可去掉的模块入口

输出：

- 关键文件列表
- 每个文件的职责
- 风险判断
- 推荐修改顺序

---

### 第二步：后台公共框架改造

只改后台通用框架层：

1. 左侧菜单扁平化
2. 顶部与主内容区层级优化
3. 搜索区卡片化
4. 表格区卡片化
5. 分页区独立化
6. 通用按钮风格统一
7. 删除/隐藏不需要的菜单入口

输出：

- 修改文件列表
- 改动摘要
- 影响范围
- 验证建议

---

### 第三步：前台公共框架改造

只改前台通用框架层：

1. 首页 banner 与轮播图优化
2. 导航统一
3. 栏目标题统一
4. 卡片模板统一
5. 搜索框统一
6. 分页统一
7. 按钮统一

输出：

- 修改文件列表
- 改动摘要
- 已统一页面范围
- 剩余页面范围

---

### 第四步：逐页细化

按模块逐页处理：

- 社区服务
- 物品信息
- 社区活动
- 资讯中心
- 意见中心
- 个人中心
- 收藏页
- 详情页
- 表单页

要求：

- 优先复用公共样式
- 少做单页特殊补丁
- 不打破原有功能路径

输出：

- 每类页面对应的修改文件
- 统一前后对比说明
- 未处理问题列表

---

### 第五步：统一收口

统一整个项目的视觉规范：

- 颜色系统
- 字体层级
- 间距系统
- 按钮系统
- 表单系统
- 卡片系统
- 圆角系统
- 阴影系统
- 图片边框系统
- 表格与分页系统

输出：

- 最终统一项清单
- 仍未统一的少量例外
- 后续维护建议

---

## 六、每一步输出格式

每次执行任务时，输出必须包含：

### 修改前
1. 本次目标
2. 预计修改文件
3. 文件职责说明
4. 为什么这是低风险修改
5. 哪些内容明确不动

### 修改后
1. 实际修改文件列表
2. 每个文件改了什么
3. 已完成目标
4. 遗留风险
5. 下一步建议

---

## 七、风险处理原则

遇到以下情况时，不要直接硬改：

- 改样式可能牵连接口逻辑
- 改模板可能影响表单提交
- 菜单隐藏可能影响权限或跳转
- 页面结构耦合严重，难以判断影响范围

处理方式：

1. 先停下
2. 先说明风险
3. 先给出低风险替代方案
4. 优先选样式层或显示层方案

---

## 八、默认审美基线

本技能默认采用以下视觉方向：

### 后台
- 简洁
- 扁平
- 清晰
- 少阴影
- 少边框
- 表格卡片化
- 操作按钮轻量化

### 前台
- 温和
- 清爽
- 社区友好
- 浅蓝白灰为主
- 点缀暖橙或柔绿
- 卡片风格统一
- 搜索与分页统一

### 前后台共同
- 字体大小统一
- 间距统一
- 圆角统一
- 按钮统一
- 表单统一
- 图片边框与阴影统一

---

## 九、完成判定

一个阶段的 UI 美化只有在以下条件同时满足时，才算达标：

1. 达到该阶段规定的视觉目标
2. 没有改动禁止修改项
3. 运行环境未变化
4. 页面功能仍可正常使用
5. 修改文件清单清晰
6. 可以平稳进入下一阶段
