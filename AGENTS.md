# AGENTS.md

## 一、项目概述

本项目是一个“社区养老服务平台”，采用前后端分目录结构：

- `back/`：后端与后台管理端相关工程，主要在 IDEA 中启动
- `front/`：前端用户界面工程，主要在 VSCode 中启动

本次任务不是重构系统，也不是更换技术栈，而是：

**在不改变现有运行环境的前提下，对前后台页面进行统一美化与风格整理。**

---

## 二、本次任务总目标

本次任务只做 UI/页面层优化，目标如下：

1. 后台管理端整体变得更简洁、扁平、现代
2. 前台用户端整体变得更清爽、柔和、统一
3. 前后台形成同一套视觉语言
4. 在尽量不影响现有功能的前提下，提高整体观感与可用性
5. 删除或隐藏不再需要的模块入口，但不擅自破坏业务代码

---

## 三、必须遵守的硬性约束

### 3.1 不允许修改的内容

严禁进行以下改动：

- 不新增任何依赖
- 不修改 `pom.xml`
- 不修改 `package.json`
- 不升级 npm
- 不更新 `package-lock.json`
- 不执行会导致依赖版本变化的安装、升级或重新解析操作
- 不升级 Spring Boot / Vue / Element / Layui / jQuery
- 不修改后端接口返回结构
- 不修改数据库结构
- 不修改 SQL 逻辑
- 不重写核心业务模块
- 不改变后端在 IDEA 中的启动方式
- 不改变前端在 VSCode 中的启动方式
- 不把项目迁移到新框架
- 不擅自修改认证、拦截器、权限校验逻辑

### 3.2 允许修改的内容

本次允许修改的范围仅限于：

- `html / jsp / vue` 页面
- `css / scss / less`
- `static` 中的图片、图标、配色资源
- 为了排版和样式所必须的小范围类名调整
- 菜单显示配置
- 与页面样式相关的少量前端交互代码
- 不影响业务逻辑的静态模板结构优化

---

## 四、推荐工作区打开方式

统一使用一个新的 VSCode 窗口，直接打开项目根目录：

`A_CommunityElderlyCareServiceSystem-main`

不要只打开 `back/` 或只打开 `front/` 作为主工作区。

原因：

1. Codex 可以同时看到前后端目录
2. 根目录下的 `AGENTS.md`、`skills/`、`specs/` 可以统一生效
3. 方便执行“先扫描项目 → 再改后台 → 再改前台 → 最后统一收口”的流程
4. 方便进行全局搜索、统一样式整理、跨目录比对

---

## 五、执行总顺序

所有工作必须严格按以下顺序推进，不得跳步大改：

1. 先扫描项目结构
2. 找出公共布局、公共样式、菜单配置、首页文件、列表页文件
3. 先改后台公共框架
4. 再改前台公共框架
5. 再做逐页细化
6. 最后统一颜色、字体、按钮、表单、卡片、边框、阴影、间距

禁止一上来对整个项目进行无计划的大面积重写。

---

## 六、统一设计方向

### 6.1 前后台统一视觉语言

统一要求：

- 主色：浅蓝 + 白 + 浅灰
- 强调色：暖橙 或 柔和绿色
- 字体大小统一
- 按钮样式统一
- 表单样式统一
- 卡片圆角统一
- 图片圆角、阴影、边框统一
- 降低厚重感，减少旧式渐变、黑边、深阴影

### 6.2 后台管理端风格目标

后台必须完成：

- 去掉厚重黑边
- 去掉深阴影
- 去掉老式渐变
- 左侧菜单改成扁平化浅色导航
- 表格页改成白底卡片 + 浅灰分隔
- 搜索区、表格区、分页区分层明确
- “详情 / 修改 / 删除”改成轻量文本按钮或小圆角按钮

### 6.3 前台用户端风格目标

前台必须完成：

- 只有首页菜单下保留轮播图
- 大图 banner 保留，但高度缩小
- 所有栏目标题统一样式
- 服务、物品、活动、资讯四类列表统一卡片模板
- 搜索框和分页组件统一一套风格
- 整体风格温和、清爽、适合社区养老服务平台

---

## 七、范围裁剪要求

本次需要去掉或隐藏以下功能入口：

- 疫情监测管理
- 物业收费管理

说明：

- 优先从菜单、页面入口、路由入口、可见按钮层面移除或隐藏
- 未经明确允许，不直接删除后端业务代码
- 未经明确允许，不删除数据库表与接口实现

## 八、工作方式要求

### 8.1 先分析，再修改

每进入一个阶段，必须先输出：

1. 准备修改哪些文件
2. 为什么这些文件需要改
3. 哪些文件明确不动
4. 改动风险点是什么

### 8.2 每次修改后必须输出

每次阶段结束后，必须输出：

1. 已修改文件列表
2. 每个文件改了什么
3. 当前阶段完成了哪些目标
4. 还有哪些风险或遗留问题
5. 下一阶段建议做什么

### 8.3 优先公共样式与公共框架

尽量优先：

- 共享布局
- 共享 CSS
- 公共按钮样式
- 公共表单样式
- 公共表格样式
- 公共卡片样式

减少在每个页面重复打补丁式修改。

## 九、阶段控制

必须使用 `specs/` 下的阶段文件推进：

- `specs/phase-00-项目扫描.spec.md`
- `specs/phase-01-后台公共框架.spec.md`
- `specs/phase-02-前台公共框架.spec.md`
- `specs/phase-03-页面细化.spec.md`
- `specs/phase-04-风格统一收口.spec.md`

规则：

- 一次只执行一个阶段
- 阶段未完成前，不进入下一阶段
- 每阶段都要有明确输出结果

## 十、技能调用规则

当任务属于以下情况时，必须参考：

`skills/ui-refresh/SKILL.md`

适用场景包括：

- 遗留项目页面美化
- 前后台统一设计语言
- 公共布局/公共样式梳理
- 不改运行环境前提下的 UI 改版
- 分阶段、低风险页面优化

---

## 十一、完成标准

一个阶段只有在以下条件同时满足时，才算完成：

1. 达到了该阶段要求的视觉目标
2. 没有改动禁止修改的文件
3. 没有改变运行环境
4. 页面仍能正常打开
5. 修改文件清单明确
6. 结果说明清晰，可继续进入下一阶段

---

## 十二、默认原则

若遇到不确定情况，默认遵守以下原则：

1. 能不动业务逻辑就不动
2. 能不动接口就不动接口
3. 能不动数据库就不动数据库
4. 能通过样式解决的问题，不通过重构解决
5. 能通过公共样式统一解决的问题，不在单页重复修补
6. 小步改动、逐步验证，优先低风险
